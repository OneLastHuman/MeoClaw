## 文档管理规则

### 1. SPEC - 产品规范
- **含义**：当前产品的准确状态，定义产品是什么样子的
- **命名**：无日期前缀
- **更新**：功能开发测试完成并 commit 后，更新 SPEC 对应文档
- **维护**：文档内部记录"最后更新时间"

### 2. PLAN - 功能计划
- **含义**：计划要做的功能设计
- **命名**：`YYYY-MM-DD-功能名.md`
- **流程**：在 PLAN 中迭代设计 → 完成后记录 CHANGELOG → 更新 SPEC

### 3. CHANGELOG - 修改记录
- **含义**：记录已完成功能的变更历史
- **命名**：`YYYY-MM-DD-变更内容.md`
- **时机**：功能完成 commit 后记录

### 4. REF - 参考资料
- **含义**：外部参考文档、技术栈文档
- **命名**：无特殊要求
- **更新**：技术栈文档已就位，一般不需频繁更新
