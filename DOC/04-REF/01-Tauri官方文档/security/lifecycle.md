# 应用程序生命周期威胁

_Source: https://v2.tauri.org.cn/security/lifecycle/_

Tauri 应用程序在生命周期的不同时间点由许多部分组成。在此我们描述经典的威胁以及您“应该”采取的应对措施。

所有这些不同的步骤将在后续章节中进行描述。

![Threat Stages During Development](/_astro/application-flow-simple.X8I9EoEI_X8esr.svg)

注意

应用程序生命周期中最薄弱的环节基本上决定了您的安全性。每一步都可能损害后续所有步骤的前提假设和完整性，因此始终着眼于全局非常重要。

## 上游威胁

标题为“上游威胁”的章节

Tauri 是您项目中的直接依赖项，我们对提交、审查、拉取请求和发布保持严格的控制。我们尽最大努力维护最新的依赖项，并采取行动进行更新或分叉修复。其他项目可能没有得到如此良好的维护，甚至可能从未经过审计。

请在集成它们时考虑其健康状况，否则您可能在不知不觉中背负了架构债务。

### 保持应用程序最新

标题为“保持应用程序最新”的章节

当您向公众发布应用时，您交付的软件包也包含了 Tauri。影响 Tauri 的漏洞可能会影响您应用程序的安全性。通过将 Tauri 更新到最新版本，您可以确保关键漏洞已得到修复，不会在您的应用程序中被利用。此外，请确保保持您的编译器 (`rustc`) 和转译器 (`nodejs`) 为最新版本，因为其中经常有安全问题得到解决。这同样适用于您的通用开发系统。

### 评估您的依赖项

标题为“评估您的依赖项”的章节

虽然 NPM 和 Crates.io 提供了许多便捷的软件包，但选择值得信赖的第三方库——或者用 Rust 重写它们——是您的责任。如果您使用受已知漏洞影响或无人维护的过时库，您的应用程序安全性和安稳的睡眠可能会受到威胁。

使用 [`npm audit`](https://docs.npmjs.net.cn/cli/v10/commands/npm-audit) 和 [`cargo audit`](https://crates.io/crates/cargo-audit) 等工具来自动化此过程，并依靠安全社区的重要工作成果。

Rust 生态系统中最近出现的趋势，如 [`cargo-vet`](https://github.com/mozilla/cargo-vet) 或 [`cargo crev`](https://github.com/crev-dev/cargo-crev)，可以帮助进一步降低供应链攻击的可能性。要了解您站在谁的肩膀上，您可以使用 [`cargo supply chain`](https://github.com/rust-secure-code/cargo-supply-chain) 工具。

我们强烈推荐的一种做法是，仅从 git 获取关键依赖项，最好使用哈希版本，其次是命名标签。这适用于 Rust 以及 Node 生态系统。

## 开发阶段威胁

标题为“开发阶段威胁”的章节

我们假设您（开发者）关心您的开发环境。确保您的操作系统、构建工具链及相关依赖项保持最新并获得合理的保护是您的责任。

我们所有人都面临的一个真正风险是所谓的“供应链攻击”，这通常被认为是针对您项目直接依赖项的攻击。然而，越来越多的真实攻击直接针对开发机器，如果能正面应对这一点，您将受益匪浅。

### 开发服务器

标题为“开发服务器”的章节

Tauri 应用程序的前端可以使用多种 Web 框架进行开发。这些框架通常都带有自己的开发服务器，通过开放端口将前端资源暴露给本地系统或网络。这使得前端能够在 WebView 或浏览器中进行热重载和调试。

在实践中，这种连接默认通常既未加密也未经过身份验证。内置的 Tauri 开发服务器也是如此，它将您的前端和资源暴露给本地网络。此外，这还允许攻击者将他们自己的前端代码推送到与攻击者处于同一网络的开发设备上。根据所暴露的功能类型，这在最坏的情况下可能导致设备受到损害。

您应该仅在可信网络上进行开发，并确保能够安全地暴露您的开发设备。如果无法做到这一点，您“必须”确保您的开发服务器使用“双向”身份验证和加密（例如 mTLS）与您的开发设备进行连接。

注意

目前内置的 Tauri 开发服务器不支持双向身份验证和传输加密，因此不应在不受信任的网络上使用。

### 加固开发环境

标题为“加固开发环境”的章节

加固开发系统取决于多种因素以及您的个人威胁模型，但我们建议遵循一些通用建议：

  * 不要将管理员账户用于编码等日常任务
  * 不要在开发机器上使用生产环境的密钥
  * 防止将密钥提交到源代码版本控制系统中
  * 使用安全硬件令牌或类似设备，以降低系统受损带来的影响
  * 保持系统更新
  * 将安装的应用程序数量保持在最低限度

您可以在 [awesome security hardening 集合](https://github.com/decalage2/awesome-security-hardening)中找到更实用的程序收集。

当然，您可以将开发环境虚拟化以抵御攻击者，但这并不能保护您免受针对您项目而非仅针对您机器的攻击。

### 确保源代码控制的认证与授权

标题为“确保源代码控制的认证与授权”的章节

如果您像大多数开发人员一样，使用源代码版本控制工具和服务提供商是开发过程中必不可少的一步。

为了确保您的源代码不会被未经授权的参与者修改，理解并正确配置源代码版本控制系统的访问控制至关重要。

此外，请考虑要求所有（常规）贡献者签署他们的提交，以防止恶意提交被归咎于未受损或非恶意的贡献者的情况。

## 构建阶段威胁

标题为“构建阶段威胁”的章节

现代组织使用 CI/CD 来生产二进制构件。

您需要完全信任这些远程（且由第三方拥有）的系统，因为它们可以访问源代码、密钥，并且能够在您无法可验证地证明生成的二进制文件与本地代码相同的情况下修改构建。这意味着，要么您信任信誉良好的提供商，要么在您自己控制的硬件上托管这些系统。

在 Tauri，我们提供了一个用于在多个平台上构建的 GitHub Workflow。如果您创建自己的 CI/CD 并依赖第三方工具，请警惕那些您未明确锁定的版本。

您应该为您发布的平台对二进制文件进行签名。虽然设置起来可能很复杂且成本较高，但终端用户期望您的应用确实来自您。

如果加密密钥已妥善存储在硬件令牌中，即使构建系统受损，也无法泄露相关的签名密钥，但可能会利用它们来签署恶意发布版本。

### 可复现构建

标题为“可复现构建”的章节

为了对抗构建阶段的后门注入，您需要实现可复现构建，以便在您本地构建或在其他独立提供商处构建时，能够验证构建资产是否完全相同。

首要问题是 Rust 默认情况下并不完全“可靠地”生成可复现构建。理论上它支持这一点，但仍存在 Bug，且最近的一次发布版本中出现了中断。

您可以在 Rust 项目的 [公共 Bug 跟踪器](https://github.com/rust-lang/rust/labels/A-reproducibility)中跟进当前状态。

您将遇到的下一个问题是，许多常见的前端打包工具也不会生成可复现的输出，因此打包的资产也可能破坏可复现构建。

这意味着您在默认情况下无法完全依赖可复现构建，遗憾的是，您需要完全信任您的构建系统。

## 分发阶段威胁

标题为“分发阶段威胁”的章节

我们已尽最大努力使应用程序的热更新变得尽可能直接和安全。然而，如果您失去了对清单服务器、构建服务器或二进制托管服务的控制，那么一切都将失控。

如果您构建自己的系统，请咨询专业的运维架构师并妥善构建它。

如果您正在寻找另一种针对 Tauri 应用的可信分发解决方案，我们的合作伙伴 CrabNebula 提供了一项服务：<https://crabnebula.dev/cloud>

## 运行阶段威胁

标题为“运行阶段威胁”的章节

我们假设 Webview 是不安全的，这导致 Tauri 在加载不受信任的用户空间内容时，针对 Webview 访问系统 API 实现了一些保护措施。

使用 [内容安全策略 (CSP)](/security/csp/) 将锁定 Webview 可以执行的通信类型。此外，[权限 (Capabilities)](/security/capabilities/) 可以防止不受信任的内容或脚本访问 Webview 内的 API。

我们还建议建立一种简单且安全的方式来报告漏洞，类似于[我们的流程](/security/#coordinated-disclosure)。
