## 问题描述

在 Response 状态时，窗口需要从 180x180 扩展到 560x580（扩大主窗口容纳气泡）。macOS 上窗口从中心缩放，导致猫在窗口缩放时跳动。

目标：**保持猫（左下角）在屏幕上的位置不变**。

---

## 环境信息

- Tauri 2.x
- macOS（Retina 显示屏，DPR=2）
- 窗口配置：`resizable: false`, `alwaysOnTop: true`, `decorations: false`, `transparent: true`

---

## 尝试过的方案

### 方案 A：Rust - set_position + set_size 顺序尝试

```rust
// 先 set_position，再 set_size
window.set_position(PhysicalPosition { x, y });
window.set_size(PhysicalSize { width, height });
```

**结果**：无效。macOS 在 set_size 时忽略之前 set_position 的设置。

---

### 方案 B：Rust - set_size 后读取实际位置，修正偏移量

```rust
window.set_size(PhysicalSize { width, height });
let new_pos = window.outer_position();
let dx = old_x - new_x;
let dy = old_bottom - new_bottom;
window.set_position(PhysicalPosition { x: new_x + dx, y: new_y + dy });
```

**结果**：无效。`outer_position()` 返回的值与实际不一致。

---

### 方案 C：Rust - 保存初始 bottom_y，恢复到该值

```rust
let bottom_y = old_pos.y + old_size.height; // 底边位置
let new_y = bottom_y - new_height;         // 目标顶边
let final_y = new_y.max(0);               // 夹紧到屏幕
window.set_position(PhysicalPosition { x: old_x, y: final_y });
window.set_size(PhysicalSize { width, height });
```

**结果**：无效。set_position 完全不生效。

---

### 方案 D：Rust - 仅测试 set_position 是否能工作

```rust
// 测试：只改位置，不改尺寸
let test_y = old_pos.y + 100;  // 向下移动 100 像素
window.set_position(PhysicalPosition { x: old_pos.x, y: test_y });
```

**结果**：set_position 调用返回成功，但位置没有变化。

---

### 方案 E：Rust - 使用 LogicalPosition vs PhysicalPosition

尝试了 `LogicalPosition` 和 `PhysicalPosition` 两种类型。

**结果**：没有区别。set_position 都不生效。

---

### 方案 F：macOS 坐标系验证

macOS 坐标系 y=0 在屏幕**顶部**，向下增加。
- 窗口顶部位置 = `outer_position.y`
- 窗口底部位置 = `outer_position.y + outer_size.height`

**验证结果**：坐标系理解正确。

---

### 方案 G：关键发现 - resizable: false 导致 set_position 无效

将 `tauri.conf.json` 中的 `resizable: false` 改为 `resizable: true` 后测试：

```rust
// resizable: true 时
window.set_position(PhysicalPosition { x: 100, y: 100 });
```

**结果**：`resizable: true` 时，`set_position` **能正常工作**！

---

### 方案 H：前端 JavaScript API（需要权限）

发现 Rust 端的 `set_position` 在 `resizable: false` 时不工作，尝试使用 Tauri JavaScript API：

```typescript
const win = getCurrentWindow();
await win.setPosition(new LogicalPosition(x, y));
```

**结果**：同样需要权限 `core:window:allow-set-position`。添加权限后，JS API 也没有生效。

---

## 根本原因

**`resizable: false` 导致 `set_position` 无法工作**。

当窗口设置为不可调整大小时，macOS 可能禁用了位置调整功能。

## 最终解决方案（2026-04-14）

### 核心思路

使用前端 JavaScript API（`@tauri-apps/api/window`）控制窗口位置，修复了 **DPR 单位转换** 问题。

### 关键修复

macOS Retina 屏幕 DPR=2 时：
- `outerPosition()` 返回 **physical 像素**
- `setPosition()` 接受 `PhysicalPosition` 或 `LogicalPosition`
- 关键：**macOS setSize 从中心扩展**，导致窗口位置偏移

### 最终修复方案（2026-04-14）

**核心思路**：先 setSize，读取实际偏移，再 setPosition 补偿。

```javascript
watch(() => animState.value, async (state) => {
  const win = getCurrentWindow();
  const dpr = await win.scaleFactor();
  const pos = await win.outerPosition(); // physical
  const catBottomPhysical = pos.y + CAT_SIZE; // 猫底部位置

  if (state === 'Response') {
    // 先 setSize（macOS 从中心扩展，顶部会上升差值的一半）
    await win.setSize(new PhysicalSize(CONTAINER_EXPANDED_W * dpr, CONTAINER_EXPANDED_H * dpr));
    // 读取新位置，计算猫底部偏移
    const newPos = await win.outerPosition();
    const actualBottom = newPos.y + CAT_SIZE;
    const dy = catBottomPhysical - actualBottom;
    if (Math.abs(dy) > 1) {
      await win.setPosition(new PhysicalPosition(newPos.x, newPos.y + dy));
    }
  } else {
    await win.setSize(new PhysicalSize(CAT_SIZE * dpr, CAT_SIZE * dpr));
    const newPos = await win.outerPosition();
    const actualBottom = newPos.y + CAT_SIZE;
    const dy = catBottomPhysical - actualBottom;
    if (Math.abs(dy) > 1) {
      await win.setPosition(new PhysicalPosition(newPos.x, newPos.y + dy));
    }
  }
});
```

### 验证结果（截图确认猫视觉位置一致）

| 状态 | 窗口 Y | 窗口高度 | 猫底部（屏幕） |
|------|--------|----------|----------------|
| idle | 249 | 180 | 249 |
| Response | 249 | 580 | 249 |
| idle 恢复 | 249 | 180 | 249 |

猫的视觉位置（右下角）在所有状态切换中完全一致。

### 必要条件

1. `tauri.conf.json` 中 `resizable: true`
2. `capabilities/default.json` 中添加 `core:window:allow-set-position`

### 测试结果

**bounds 数据（logical 像素）**：
| 状态 | X | Y | W | H | 底部(Y+H) |
|------|---|---|---|---|-----------|
| idle | 870 | 249 | 180 | 180 | 429 |
| Response | 870 | 249 | 540 | 580 | 829 |
| idle 恢复 | 870 | 249 | 180 | 180 | 429 |

**bounds 数据分析**：
- 窗口 Y=249 在所有状态下一致
- 但 Response 时窗口变高（580 vs 180）
- 猫底部 = Y + 猫高度(180)，所以猫底部应该始终在 Y=249

**截图验证**：
- idle 截图：360x360 像素（physical，180x2）
- Response 截图：1080x1160 像素（physical，540x2）
- **视觉对比**：Response 时猫的眼睛位置明显更靠下

### 问题分析

bounds 数据看起来正确，但截图显示猫的位置不对。可能的原因：

1. **截图方式问题**：macOS 截图可能使用窗口坐标而非屏幕坐标
2. **DPR 计算问题**：bounds 可能是 logical，但截图是 physical
3. **锚点参考点问题**：macOS setSize 的锚点可能不是我们理解的"中心"

### 仍存在的问题

- **bounds Y=249 一致**，但 **猫的视觉位置在 Response 时偏下**
- 需要进一步排查截图获取方式和实际显示位置的关系
