# 窗口拖拽状态实现（2026-04-18）

## 这次实现解决什么问题

贴边半隐藏设计草案里已经把“动画状态”和“窗口行为状态”拆开了，但代码里之前只有动画状态，没有真正落地的窗口拖拽状态。

这次先完成第一阶段最小闭环：

- 新增窗口行为状态 `normal / dragging`
- 用户按住猫本体拖动窗口时，进入 `dragging`
- 系统拖拽结束后，回到 `normal`
- `dragging` 状态挂接占位精灵图 `public/anim/edge-placeholders/dragging.png`
- 不改动现有 `AnimState`，避免把贴边逻辑塞进动画状态机

---

## 当前行为

### 触发条件

在 `src/components/DesktopPet.vue` 中：

- 鼠标左键按下
- 点击位置落在猫本体区域内
- 且这次操作不是双击切换输入框

满足后进入前端自定义拖拽流程，同时把窗口行为状态切到 `dragging`。

### 拖拽期间表现

- `windowBehaviorState` 从 `normal` 切到 `dragging`
- 根节点写入 `data-window-behavior="dragging"`
- 根节点和子元素统一使用 `grabbing` 光标
- 当前显示动画临时切换为占位精灵图 `dragging.png`
- 猫本体保留轻微亮度/饱和度变化，作为额外即时反馈

### 拖拽结束

指针释放或取消后，会统一恢复：

- `windowBehaviorState = normal`

这样后续接入吸附、半隐藏、探出逻辑时，可以继续从这个状态出口往下扩展。

## 实现要点

### 1. 窗口行为状态独立于动画状态

当前仍然保留现有动画状态：

- `idle`
- `working`
- `Response`
- `EnterInput`
- 其他已有动画态

拖拽状态不写进 `AnimState`，而是单独维护：

```ts
const windowBehaviorState = ref<WindowBehaviorState>('normal');
```

这样可以避免后续出现：

- `idleDragging`
- `ResponseDragging`
- `workingDragging`

这一类组合爆炸。

### 2. 拖拽动画通过窗口行为状态覆盖当前显示动画

实现方式：

- 不新增 `AnimState.dragging`
- 保持业务动画状态仍由 `AnimState` 驱动
- 当 `windowBehaviorState === dragging` 时，显示动画临时改用 `dragging.png`
- 拖拽结束后自动恢复到底层原本的 `idle / working / Response ...` 动画

同时保留日志：

- `[WINDOW] behavior normal -> dragging`
- `[WINDOW] behavior dragging -> normal`

### 3. 拖拽入口统一封装

当前拖拽由 `beginWindowDrag / onPointerMove / finishWindowDrag` 串起来：

- 按下时记录当前窗口缩放和窗口尺寸
- 移动超过阈值后切换到 `dragging`
- 每次移动时用 `setPosition()` 更新窗口位置
- 松手或取消后统一恢复 `normal`

补充约束：

- 即使当前业务状态是 `Response / working`，进入 `dragging` 后也统一按拖拽窗口尺寸 `180x180 * scale` 计算中心点
- 不再沿用按下瞬间的大窗口半宽半高，避免 `Response` 切进 `dragging` 时鼠标中心偏移

### 4. 拖拽定位规则

当前实现不是让鼠标吸附窗口左上角，而是让窗口中心点跟随鼠标：

- `targetX = 鼠标屏幕坐标X - 窗口半宽`
- `targetY = 鼠标屏幕坐标Y - 窗口半高`

这样拖动时鼠标会始终落在桌宠窗口中心，更符合“抓住猫本体中心移动”的手感。

### 5. 这次还没有做的内容

这次没有实现：

- 松手后的左右边缘吸附
- `edgeHiddenLeft / edgeHiddenRight`
- `edgePeekLeft / edgePeekRight`
- 隐藏态消息提醒
- 隐藏态下 Response / working 气泡裁剪逻辑

这部分会直接复用当前的窗口行为状态骨架继续往下接。
