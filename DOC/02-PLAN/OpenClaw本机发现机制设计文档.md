# OpenClaw 本机发现机制设计方案 v2

> 创建时间：2026/04/23
> 状态：待评审
> 版本：v2（改进自初版）

---

## 一、这个功能解决什么问题？

**用户遇到的不便：**

现在桌宠只能连接写死的地址 `127.0.0.1:18789`。如果：
- OpenClaw 用的是开发模式（端口 19001）→ 连不上
- OpenClaw 还没启动 → 桌宠一直尝试连接但不做任何提示

**有了这个功能：**

桌宠会自动找出 OpenClaw 在哪里，不需要你改任何配置。如果找不到，会清清楚楚告诉你"OpenClaw 没运行"，而不是傻等。

---

## 二、设计原则

**唯一真理：端口连通才是真连通**

不管配置文件怎么写、进程有没有在跑，最终只有"端口能连上"才能证明 OpenClaw 真的可用。

---

## 三、连接流程

```
桌宠启动
    │
    ▼
┌─────────────────┐
│ 第1步：读配置文件  │ ← 可选，用来获取"连接地址提示"
│  ~/.openclaw/   │
│  openclaw.json   │
└────────┬────────┘
         │ 拿到 host 提示
         ▼
┌─────────────────┐
│ 第2步：端口探测   │ ← 核心：试试端口能不能连上
│  18789 → 19001  │
└────────┬────────┘
         │ 能连上
         ▼
┌─────────────────┐
│ 第3步：建立连接   │
│  WebSocket      │
└────────┬────────┘
         │
         ▼
    连接成功 or 失败提示
```

---

## 四、每一步做什么

### 第1步：读配置文件（获取提示）

**目的**：知道应该尝试连接哪个地址

读取 `~/.openclaw/openclaw.json`，看 `gateway.bind` 字段：

| 配置值 | 桌宠尝试连接的地址 |
|--------|-------------------|
| `loopback`（默认） | 127.0.0.1 |
| `auto` | 127.0.0.1 |
| `lan` | 127.0.0.1 |
| `non-loopback` | 127.0.0.1 |
| `custom` | 127.0.0.1 |
| 文件不存在 | 127.0.0.1（默认） |

> **为什么要这样做？**
>
> OpenClaw 配置 `lan` 或 `custom` 时，意味着它允许局域网访问。但桌宠在本机运行，所以还是连 `127.0.0.1`。
>
> 配置文件只给一个"提示"，真正能不能连上，由端口探测决定。

### 第2步：端口探测（核心）

**目的**：确认 OpenClaw 真的可以连接

按顺序尝试这两个端口，每个给 3 秒超时：

1. `http://127.0.0.1:18789`（默认端口）
2. `http://127.0.0.1:19001`（开发模式端口）

**探测方法：**
1. TCP 连接测试（3秒超时）
2. HTTP GET 健康检测
3. 检查是否返回成功

只要有一个端口通了，就用那个端口。

### 第3步：建立连接

用探测到的地址和端口，建立 WebSocket 连接。

## 五、连接状态

桌宠会持续显示当前 OpenClaw 的连接状态：

| 状态 | 显示 |
|------|------|
| 正在探测 | "正在寻找 OpenClaw..." |
| 已连接 | "已连接 OpenClaw" |
| 未检测到 | "未检测到 OpenClaw，请确保它正在运行" |
| 连接断开 | "OpenClaw 连接断开，正在重连..." |
| 重连失败 | "无法连接 OpenClaw，请检查是否运行中" |

---

## 六、自动重连

```
连接成功
    │
    ▼
持续监听 WebSocket 连接状态
    │
    ├── 收到断开事件
    │
    ▼
显示"正在重连..."
    │
    ▼
重新执行端口探测（第1步 + 第2步）
    │
    ▼
再次尝试连接
```

重连会一直尝试，直到连上为止。

## 七、技术实现

### 新增文件

| 文件 | 做什么 |
|------|--------|
| `src-tauri/src/backend/openclaw_discovery.rs` | 所有发现逻辑 |

### 修改文件

| 文件 | 修改内容 |
|------|----------|
| `src-tauri/src/backend/openclaw_client.rs` | 不再硬编码地址，改用发现机制 |
| `src-tauri/src/backend/manager.rs` | 集成发现流程，管理连接状态 |
| `src-tauri/src/lib.rs` | 启动时触发发现 |

### 核心结构

```rust
// 发现结果
struct DiscoveryResult {
    host: String,        // "127.0.0.1"
    port: u16,           // 18789 或 19001
}

// 连接状态
enum ConnectionState {
    Idle,                    // 还未开始
    Discovering,             // 正在寻找
    Connected,               // 已连接
    Disconnected,            // 断开（等待重连）
    Failed(String),           // 彻底失败（给出原因）
}
```

### 端口探测代码逻辑

```rust
async fn probe_port(host: &str, port: u16) -> bool {
    // 1. TCP 连接测试（3秒超时）
    // 2. HTTP GET 健康检测
    // 3. 检查响应
}

async fn discover() -> Option<DiscoveryResult> {
    // 先从配置文件读取 host（默认 127.0.0.1）
    let host = read_config_host().unwrap_or("127.0.0.1");

    // 探测端口：18789 → 19001
    for port in [18789, 19001] {
        if probe_port(&host, port).await {
            return Some(DiscoveryResult { host, port });
        }
    }

    None  // 全部失败
}
```

## 八、和原来方案的对比

| 方面 | 原来 | 现在 |
|------|------|------|
| 发现层次 | 3层（配置→进程→端口） | 2层（配置→端口） |
| 进程扫描 | 有（容易误判） | 去掉 |
| 配置解析 | 复杂（5种值各不同） | 简化（只取host提示） |
| 失败处理 | 静默返回默认值 | 明确提示用户 |
| 重连机制 | 没提 | 自动重连 |

### 为什么去掉进程扫描？

原方案用 `pgrep -f openclaw` 检查进程，但这样做有问题：

- 你在终端输入 `ps aux | grep openclaw` 查看进程，桌宠会以为 OpenClaw 在运行
- Windows 上搜 `*node*` 太宽泛，可能匹配其他程序

**端口能连上才是真连上** — 进程在不在，只有端口知道。

## 九、上线后的用户体验

| 场景 | 桌宠表现 |
|------|----------|
| OpenClaw 正常运行 | 显示"已连接 OpenClaw"，正常工作 |
| OpenClaw 没启动 | 显示"未检测到 OpenClaw，请确保它正在运行" |
| OpenClaw 用 dev 模式（19001） | 自动探测到 19001，正常连接 |
| OpenClaw 突然崩溃 | 显示"连接断开，正在重连..."，自动等待恢复 |
| OpenClaw 重启后恢复 | 自动重连，恢复正常 |
