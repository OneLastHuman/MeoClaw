# OpenClaw 本机发现机制设计方案 v2

> 创建时间：2026/04/23
> 状态：✅ 已实现（2026/04/24）
> 版本：v2（基于调研修正）

---

## 一、这个功能解决什么问题？

**用户遇到的不便：**

现在桌宠只能连接写死的地址 `127.0.0.1:18789`。如果：
- OpenClaw 用的是 dev 模式（端口 19001）→ 连不上
- OpenClaw 还没启动 → 桌宠一直尝试连接但不做任何提示

**有了这个功能：**

桌宠会自动找出 OpenClaw 在哪里，不需要你改任何配置。如果找不到，会清清楚楚告诉你"OpenClaw 没运行"，而不是傻等。

---

## 二、设计原则

**唯一真理：端口连通才是真连通**

不管配置怎么写、进程有没有在跑，最终只有"端口能连上"才能证明 OpenClaw 真的可用。

---

## 三、发现机制范围

本机制只负责发现 **OpenClaw Gateway**。

**OpenClaw 和 Hermes 是两个独立的系统：**

| 系统 | 是什么 | 发现方式 |
|------|--------|----------|
| OpenClaw | Gateway（类似路由器） | 端口探测（本次方案） |
| Hermes | AI 服务（大脑） | mDNS + 配置文件（已存在） |

两者互不影响，各自有独立的存在理由。

---

## 四、连接流程

```
桌宠启动
    │
    ▼
┌─────────────────┐
│  端口探测        │ ← 核心：试试端口能不能连上
│  18789 → 19001  │
└────────┬────────┘
         │ 能连上
         ▼
┌─────────────────┐
│  建立连接        │
│  WebSocket      │
└────────┬────────┘
         │
         ▼
    连接成功 or 失败提示
```

---

## 五、每一步做什么

### 端口探测（核心）

**目的**：确认 OpenClaw 真的可以连接

按顺序尝试这两个端口，每个给 3 秒超时：

1. `ws://127.0.0.1:18789`（默认端口）
2. `ws://127.0.0.1:19001`（dev 模式端口）

**探测方法：**
1. TCP 连接测试（3秒超时）
2. WebSocket 连接尝试
3. 检查是否建立成功

只要有一个端口通了，就用那个端口。

**为什么不需要读配置文件？**

OpenClaw 的配置文件中 `gateway.bind` 是枚举值（loopback/lan/non-loopback），用于决定 Gateway 绑定到哪个网络接口。但桌宠在本机运行，**总是连接 127.0.0.1**，跟配置无关。

---

## 六、连接状态

桌宠会持续显示当前 OpenClaw 的连接状态：

| 状态 | 显示 |
|------|------|
| 正在探测 | "正在寻找 OpenClaw..." |
| 已连接 | "已连接 OpenClaw" |
| 未检测到 | "未检测到 OpenClaw，请确保它正在运行" |
| 连接断开 | "OpenClaw 连接断开，正在重连..." |
| 重连失败 | "无法连接 OpenClaw，请检查是否运行中" |

---

## 七、自动重连

桌宠**已经有重连机制**（指数退避，最多 10 次）。

这次只需要增强：重连时**重新探测端口**（因为用户可能从 18789 切换到了 19001，或者反过来）。

```
连接成功
    │
    ▼
持续监听 WebSocket 连接状态
    │
    ├── 收到断开事件
    │
    ▼
显示"正在重连..."
    │
    ▼
重新执行端口探测（18789 → 19001）
    │
    ▼
再次尝试连接
```

重连会一直尝试，直到连上为止。

## 八、技术实现

### 修改文件

| 文件 | 修改内容 |
|------|----------|

| `src-tauri/src/lib.rs` | 启动时触发发现 |

### 核心结构

```rust
// 发现结果
struct DiscoveryResult {
    host: String,        // "127.0.0.1"
    port: u16,           // 18789 或 19001
    url: String,         // "ws://127.0.0.1:18789"
}

// 连接状态
enum ConnectionState {
    Idle,                    // 还未开始
    Discovering,             // 正在寻找
    Connected,               // 已连接
    Disconnected,            // 断开（等待重连）
    Failed(String),          // 彻底失败（给出原因）
}
```

### 端口探测代码逻辑

```rust
async fn probe_port(host: &str, port: u16) -> bool {
    // 1. TCP 连接测试（3秒超时）
    // 2. WebSocket 连接尝试
}

async fn discover() -> Option<DiscoveryResult> {
    let host = "127.0.0.1";  // 总是本机

    // 探测端口：18789 → 19001
    for port in [18789, 19001] {
        if probe_port(&host, port).await {
            return Some(DiscoveryResult {
                host: host.to_string(),
                port,
                url: format!("ws://{}:{}", host, port),
            });
        }
    }

    None  // 全部失败
}
```

## 九、和原来方案的对比

| 方面 | 原方案 v1 | 现在 v2 |
|------|-----------|---------|
| 配置读取 | 读取 gateway.bind 获取 host 提示 | 不读配置，直接用 127.0.0.1 |
| 端口探测 | 18789 → 19001 | 不变 |
| 进程扫描 | 有（容易误判） | 去掉 |
| 重连机制 | 没提 | 复用现有重连逻辑 |

### 为什么去掉配置读取？

调研发现 `gateway.bind` 是网络接口配置，不是 host 配置。桌宠总是连本机，这个值对桌宠没有意义。

### 为什么去掉进程扫描？

原方案用 `pgrep` 检查进程，但这样做有问题：
- 搜 `openclaw` 可能匹配到其他东西
- Windows 上更复杂

**端口能连上才是真连上** — 进程在不在，只有端口知道。

## 十、上线后的用户体验

| 场景 | 桌宠表现 |
|------|----------|
| OpenClaw 正常运行 | 显示"已连接 OpenClaw"，正常工作 |
| OpenClaw 没启动 | 显示"未检测到 OpenClaw，请确保它正在运行" |
| OpenClaw 用 dev 模式（19001） | 自动探测到 19001，正常连接 |
| OpenClaw 突然崩溃 | 显示"连接断开，正在重连..."，自动等待恢复 |
| OpenClaw 重启后恢复 | 自动重连，恢复正常 |
