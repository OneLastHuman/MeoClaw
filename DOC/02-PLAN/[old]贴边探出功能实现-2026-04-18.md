## 关键实现

### 1. 露出与热区全部改成“身位比例”

本次把 `peek` 相关规则从固定像素改成运行时比例换算：

- hidden 露出量：`65% 身位`
- peek 露出量：`1/2 身位`
- peek 内侧缓冲：`1/4 身位`

代码仍然在运行时换算为像素，但规则口径已经统一成比例，缩放后会自动跟随窗口宽度变化。

这里额外记录一个修正：

- 早期版本把 `peek` 总露出写成了 `1/3 身位`
- 在 180 宽窗口下，这只会比 hidden 多露出 `15px`
- 视觉上几乎等于“没有探出”

因此本次改成 `1/2 身位`，保证 hidden -> peek 肉眼可见。

### 2. 用全局鼠标坐标驱动 hidden / peek 切换

这轮没有继续依赖组件本身的 `mouseenter / mouseleave`，而是改用 Tauri 的 `cursorPosition()` 轮询当前鼠标物理坐标，再结合显示器工作区边界做判断。

这样做的原因是：

- hidden 状态下窗口只有一小段露在屏幕内
- peek 需要“露出区域 + 向屏幕内侧缓冲”这类屏幕坐标热区
- 只靠 DOM hover 无法稳定表达这类逻辑

实际规则：

- hidden 状态下，鼠标进入边缘露出热区就触发 `peek`
- peek 状态下，只要鼠标还在“露出区域 + 内侧缓冲”内，就保持探出
- 鼠标离开后立即开始缩回 hidden 的位移动画
- 当前不再保留额外等待时间

这里也补一条关键修正：

- 早期版本在 hidden / peek 判断里调用的是 `currentMonitor()`
- 但窗口本来就被故意放到屏幕外一部分，Tauri 文档明确说明“如果无法检测到当前显示器，则返回 null”
- 一旦这里拿到 `null`，整个 `peek` 逻辑就会提前退出

所以本次改成：

1. 优先使用 `cursorPosition()` + `monitorFromPoint()` 获取当前鼠标所在显示器
2. 进入 hidden 时缓存吸附命中的 monitor 边界
3. 如果临时取不到 monitor，就回退到缓存的边界继续判断

这样 hidden / peek 不会因为窗口部分离屏而失效。

另外，这轮还修了一次“运行一段时间后 `peek` 失效”的稳定性问题。

反思下来，根因不只是轮询频率，而是实现原语选错了。早期实现里：

- `peek` 鼠标跟踪使用 `requestAnimationFrame` 长期循环
- `peek` / 吸附窗口位移动画本身也使用 `requestAnimationFrame`
- 每一帧都重复调用多次 Tauri IPC 查询鼠标、窗口和显示器

这会带来两个风险：

1. WebView 长时间运行后，`requestAnimationFrame` 可能被节流，跟踪不再稳定。
2. 如果窗口位移动画在节流或异常场景里没有正常结束，`peekTransitionInFlight` 这类保护标记可能长时间不复位，之后所有 `peek` 都会被短路。
3. 高频 IPC 中如果某一次查询卡住，跟踪中的 in-flight 标记会长期占住，看起来就像 `peek` 永久失效。

这次改成：

- 长期跟踪使用轻量定时器
- 原生窗口位移动画改成时间驱动的定时器步进，不再依赖浏览器渲染帧
- 贴边后的 monitor / window 上下文尽量缓存，不再每次循环都重新查完整窗口信息
- 鼠标位置查询增加超时保护，避免单次卡住拖死整条 `peek` 链路
- 每一步 `setPosition()` 也加超时保护，避免原生调用卡住后把动画状态永久锁死

再往后排查时，确认了另一个更贴近真实使用的根因：

- 当用户刚刚点击过其他软件、鼠标焦点还在别的软件上时
- 桌宠窗口内的 `pointerenter / pointermove / pointerleave` 事件并不可靠
- 如果 `peek` 只依赖窗口事件，就会表现成“桌宠贴边后不再探出”

这也是为什么之前看起来像是 “1/2 身位”“1 身位” 之类的位置问题，但实际根因并不是热区比例本身，而是事件链在非激活窗口场景下失效。

因此这次最终调整成双通道方案：

- 保留窗口内事件，作为有焦点时的即时响应
- hidden / peek 状态下再加一个基于 `cursorPosition()` 的全局鼠标兜底检测
- 兜底判断使用实时鼠标位置、窗口位置、窗口尺寸和当前 monitor 边界，不要求桌宠先拿到焦点

这样即使用户刚操作过其他软件，只要鼠标移到贴边露出的桌宠区域，`peek` 仍然可以正常触发。

### 3. `peek` 仍属于紧凑窗口行为

这次顺手修正了一个结构性问题：

- 之前只有 `edgeHidden*` 被当成紧凑窗口
- `dragging` 和 `edgePeek*` 没有完整纳入同一套紧凑尺寸/UI 约束

现在统一成：

- `normal` 之外的窗口行为状态都走紧凑窗口尺寸
- `dragging / hidden / peek` 都不显示输入框、工作气泡、响应气泡、文件覆盖层

这样可以避免：

- 进入 `peek` 时窗口尺寸突然切回完整模式
- 从隐藏态开始拖拽时错误展开大窗口

### 4. 动画与位移同步

`peek` 状态不是只换图，也不是只移动窗口，而是同步执行：

1. 切到 `edgePeekLeft / edgePeekRight` 动画
2. 维持紧凑尺寸
3. 用 `200ms` 平滑位移动画把窗口向屏幕内侧移动
4. 缩回时再滑回 hidden 对应位置

窗口底边锚点规则保持不变，没有改动既有的尺寸补偿逻辑。
